Concept · Event Sourcing

Why CQRS splits your reads from your writes

A mental model for the pattern every cpp-context-* service is built on — and the one idea that makes the rest of the architecture click.

1 The core idea

In a traditional service, one model both accepts changes and answers questions. CQRSCommand Query Responsibility Segregation — separate the write model (commands that change state) from the read model (queries that return it). says: stop forcing one shape to do both jobs. Commands go through a write side optimised for validity; queries hit a read side optimised for speed.

Commands and queries orbit the same domain but never share a path. An event is the only thing that crosses between them — the write side emits it, the read side rebuilds a viewstoreViewstore — a denormalised read database, rebuilt by replaying events. Queries hit this, never the event store. from it.

2 An analogy

📒
Think of a bank ledger
You never edit a past balance. You append a transaction (a command → event), and your "current balance" (the view) is just a running total derived from the log. Lose the balance? Replay the ledger and it's back.

That append-only log is your event store. The "current balance" page your teller sees is the viewstore. They are deliberately different shapes.

3 Why it's worth the cost

CQRS adds moving parts, so it earns its keep only when reads and writes have genuinely different needs — which they do in a case-management platform.

⚠️
The trap to avoid
Don't query the write model "just this once" inside a command handler. That couples the sides back together and is the anti-pattern most CPP review comments flag.
