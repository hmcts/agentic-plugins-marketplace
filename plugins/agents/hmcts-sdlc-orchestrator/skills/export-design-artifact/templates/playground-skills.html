Playground Skills

A playground is a single interactive HTML file: controls on one side, a live preview on the other, and a generated prompt at the bottom with a copy button. You explore visually, then paste the prompt back to Claude. The output isn't the page — it's the instruction it writes for you.

03 What's available — the 6 types

design-playground

Visual design

Components, layouts, spacing, colour, typography — anything you'd rather see than describe.
Best for: dialling in a card/button, picking a palette, tuning a layout before asking for code.
data-explorer

Data & queries

Build SQL, API calls, pipelines or regex with live results from sample inputs.
Best for: shaping a query, testing a regex, designing an API request without round-tripping.
concept-map

Learning & scope

Concept maps, knowledge-gap explorers, scope maps — navigate a topic and mark what to dig into.
Best for: onboarding to a domain, scoping a feature, surfacing what you don't yet know.
document-critique

Document review

Inline suggestions with an approve / reject / comment workflow over a draft.
Best for: triaging edits on a spec, RFC or README — decide per-suggestion, emit a clean instruction set.
diff-review

Code review

Git diffs, commits or PRs with line-by-line commenting.
Best for: walking a diff and attaching comments, then handing the consolidated review back to Claude.
code-map

Codebase architecture

Component relationships, data flow and layer diagrams of a codebase you can explore.
Best for: understanding an unfamiliar service (e.g. a cpp-context-*), picking where a change lands.

If a topic doesn't fit cleanly, the skill adapts the closest type. Every type still obeys the same rules: single file, live preview, copyable prompt, dark theme, 3–5 presets.

04 Playground or static template?

These complement the theme gallery. The distinction is read vs. manipulate:

Reach for a playground

  • The input space is large, visual or structural
  • You'll iterate — try, see, adjust, repeat
  • The goal is to produce a better prompt
  • "I'll know it when I see it"

Reach for a static template

  • The content is settled and meant to be read
  • It's a deliverable: report, plan, ADR, dashboard
  • You want it shared/archived, not tweaked
  • No decision loop — just communicate
