ADR-0042

Choosing the read store for reservation queries

Accepted

1. Context

The new reservation flow needs a read model that survives event replay and serves slot-availability queries under listing peak load. We evaluated three options against four weighted criteria: query speed, operational cost, replay fit, and team familiarity.

4. Trade-offs of the leading option

Pros

  • Already the platform's standard viewstore pattern
  • Liquibase-managed, replayable schema
  • No new infra to operate

Cons

  • Free-text search weaker than ES
  • Needs index tuning at peak

5. Decision

PostgreSQL viewstore
Adopt the standard PostgreSQL viewstore rebuilt by the event listener. Revisit Elasticsearch only if free-text slot search becomes a product requirement. Redis is rejected — it can't be rebuilt from events alone.
